🧪 Test Suites

A2Apex provides comprehensive test suites covering all aspects of A2A protocol compliance. Each suite tests a specific domain and provides detailed pass/fail results with diagnostic information.

Test Suite Overview

The full compliance suite consists of five test categories:

Suite Endpoint Description
⚡ Live Endpoint /api/live-test Core A2A endpoint testing
🔒 Auth & Security /api/test/auth Authentication scheme validation
❌ Error Handling /api/test/errors JSON-RPC 2.0 error compliance
📡 Streaming /api/test/streaming SSE event format and ordering
⏱️ Performance /api/test/performance Latency and concurrency tests

⚡ Live Endpoint Tests

Tests the core A2A protocol endpoints by making real JSON-RPC calls to your agent.

⚡ Live Endpoint Tests

  • agent_card_fetch Agent Card accessible at /.well-known/agent-card.json
  • message_send message/send JSON-RPC method returns valid task
  • task_get tasks/get returns task by ID
  • task_cancel tasks/cancel handles cancel request correctly
  • invalid_method Returns -32601 for unknown methods
  • task_lifecycle Task state transitions follow valid paths

Running Live Tests

from a2apex import A2ApexClient

client = A2ApexClient()
results = client.test_agent("https://your-agent.example.com")

for test in results:
    status = "✅" if test.passed else "❌"
    print(f"{status} {test.name}: {test.message}")

🔒 Auth & Security Tests

Validates authentication schemes declared in the Agent Card and tests security enforcement.

🔒 Auth & Security Tests

  • security_schemes_declared Agent Card declares security schemes
  • apikey_scheme_valid API key scheme has required fields (name, in)
  • http_bearer_valid HTTP bearer scheme is correctly configured
  • oauth2_flows_valid OAuth2 flows have required URLs and scopes
  • unauthenticated_rejected Requests without auth are rejected (if auth required)
  • invalid_token_rejected Invalid tokens return 401
  • https_enforcement Production URL uses HTTPS

Running Auth Tests

curl -X POST https://api.a2apex.io/api/test/auth \
  -H "Content-Type: application/json" \
  -d '{"agent_url": "https://your-agent.example.com"}'

❌ Error Handling Tests

Tests that your agent returns proper JSON-RPC 2.0 error responses for various error conditions.

❌ Error Handling Tests

  • malformed_json Returns -32700 Parse Error for invalid JSON
  • invalid_jsonrpc Returns -32600 Invalid Request for bad JSON-RPC
  • method_not_found Returns -32601 for unknown methods
  • invalid_params Returns -32602 for invalid parameters
  • task_not_found Returns -32001 for non-existent task
  • task_not_cancelable Returns -32002 for completed tasks
  • oversized_payload Rejects payloads exceeding limits

JSON-RPC Error Codes

Code Name Description
-32700Parse ErrorInvalid JSON
-32600Invalid RequestInvalid JSON-RPC
-32601Method Not FoundUnknown method
-32602Invalid ParamsInvalid parameters
-32603Internal ErrorServer error
-32001Task Not FoundA2A: Task doesn't exist
-32002Task Not CancelableA2A: Task already complete
-32003Push Not SupportedA2A: No push capability
-32004Unsupported OperationA2A: Operation not supported

📡 Streaming Tests

Tests Server-Sent Events (SSE) streaming functionality for agents that declare streaming: true.

📡 Streaming Tests

  • sse_connection SSE endpoint accepts connection
  • event_format Events have correct event: and data: format
  • state_transitions State changes follow valid transition order
  • artifact_streaming Artifacts stream with proper chunking
  • stream_termination Stream ends on terminal state
  • timeout_handling Stream handles timeout gracefully

Expected SSE Event Format

event: task_status
data: {"status": {"state": "working"}, "taskId": "abc123"}

event: task_artifact
data: {"artifactId": "art1", "parts": [{"kind": "text", "text": "Hello..."}]}

event: task_status  
data: {"status": {"state": "completed"}, "taskId": "abc123"}

⏱️ Performance Tests

Measures agent response times, concurrency handling, and performance characteristics.

⏱️ Performance Tests

  • agent_card_latency Agent Card fetch time (should be <500ms)
  • message_send_latency Initial message/send response time
  • concurrent_requests Handles 5+ concurrent requests
  • task_isolation Concurrent tasks don't interfere
  • cold_start First request latency (cold start)
  • warm_requests Subsequent request latency (warm)

State Machine Validation

A2A defines a strict state machine for task lifecycle. A2Apex validates that your agent follows valid state transitions.

Valid Task States

State Type Description
submittedInitialTask received, processing starting
workingIn ProgressAgent actively working
input-requiredInterruptedWaiting for user input
auth-requiredInterruptedNeeds additional auth
completedTerminalSuccessfully finished
failedTerminalFailed with error
canceledTerminalCanceled by request
rejectedTerminalAgent refused task

Valid Transitions

submitted → working, input-required, auth-required, completed, failed, rejected, canceled
working → input-required, auth-required, completed, failed, canceled
input-required → working, completed, failed, canceled
auth-required → working, completed, failed, canceled
⚠️ Terminal States

Once a task reaches a terminal state (completed, failed, canceled, rejected), no further transitions are allowed.
